# Release Notes: v1.5.6-beta4

Release date: 2026-06-02

## Summary

`v1.5.6-beta4` is a security and static-analysis hardening release. It drives the
backend static-analysis surface to zero findings, migrates deprecated APIs,
hardens several authentication, SSRF, and audit boundaries, and converts five
previously deferred expected-failure (XFAIL) tests into enforced contracts.

There are no schema migrations and no UI feature changes. The one behavior change
that external integrators must note is the new APIv2 authentication status code
(see Security Notes).

## Highlights

- **Static-analysis clean.** `staticcheck`, `golangci-lint`, and `gosec` now
  report zero findings across the codebase; `go vet` (with nilness), the full
  `go test ./...` suite (including `-race`), `govulncheck`, and the frontend
  lint/typecheck/unit suites are all green.
- **Deprecations removed.** Migrated `sagernet/sing/common/atomic.Int64` to the
  standard library `sync/atomic`, and replaced the deprecated
  `net.Error.Temporary()` connection check with a timeout-based equivalent.
- **Dead code and error handling.** Removed unused functions, variables, and
  parameters; simplified redundant conditions; and made previously unchecked
  error returns explicit (HTTP response writes, database migrations, session
  save, app start, and best-effort connection closes).

## Security Notes

- **APIv2 authentication status.** An invalid or expired bearer token on
  `/apiv2/*` now returns HTTP `401 Unauthorized` instead of HTTP `200` with a
  `success:false` body. The browser UI is unaffected because it authenticates
  with cookie sessions on `/api`, not bearer tokens on `/apiv2`. External API
  consumers that previously inspected the `200` body must now check the HTTP
  status code.
- **Strict session cookies (opt-in).** A new `sessionSameSiteStrict` setting
  (default `false`) issues session cookies with `SameSite=Strict` when enabled.
  The default remains `SameSite=Lax`, so there is no change unless you turn it on.
- **SSRF hardening.**
  - Telegram proxy URLs now reject embedded credentials (`user:pass@host`
    userinfo); use the dedicated proxy username/password fields instead.
  - Optional HTTP URL settings now reject private, loopback, link-local, and
    multicast IP-literal hosts (DNS is not resolved, so public hostnames are
    unaffected).
- **Constant-time token comparison.** The API-token-scope comparison no longer
  folds the length difference through a single byte, so a length mismatch that
  is an exact multiple of 256 can no longer be masked.

## Audit

- Successful settings saves now emit a `settings_save_succeeded` audit event
  (info severity, no secret values), complementing the existing
  `settings_save_rejected*` events.

## Backup / Restore

- A backup → restore round-trip now strictly preserves backup-table row counts,
  including the `tls` no-TLS sentinel row. The corresponding regression guard was
  converted from an expected-failure to an enforced assertion.

## Upgrade Notes

- No manual migration step is required; existing administrators, settings,
  clients, tokens, and sing-box config are preserved.
- If you integrate with `/apiv2` using bearer tokens, update your client to treat
  HTTP `401` (not a `200` body) as the authentication-failure signal.
- The new `sessionSameSiteStrict` setting defaults to off; enable it only if your
  panel is served on a single site and you do not rely on cross-site cookie
  delivery.
